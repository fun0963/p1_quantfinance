# 量化交易系統 — 完整任務拆解(最小可執行單位)

> **前提**:台股 + 美股 + 期貨/選擇權|從零開始|Python 為主
> **閱讀方式**:每個模組列出「任務編號|任務|完成目標|注意事項」。任務粒度以「一個人 0.5~3 天可完成」為原則。
> **建議開發順序**:M0 → M1 → M2 → M8(paper trading 骨架)→ M5 → M6 → M4 → M7 → M3 → M9 → M10 → M11

---

## M0. 基礎建設與開發環境(所有模組的地基)

| # | 任務 | 完成目標 | 注意事項 |
|---|------|---------|---------|
| 0.1 | 建立 Git repo 與專案結構 | monorepo 或多 repo 決策完成;目錄結構定案(`data/`, `research/`, `execution/`, `risk/`, `ops/`) | 從第一天就用版本控制;`.gitignore` 排除資料檔與 API key |
| 0.2 | Python 環境管理 | 用 `uv` 或 `poetry` 鎖定依賴;Python 3.11+ | 台股套件(如 shioaji)與科學計算套件版本衝突常見,先建 lock file |
| 0.3 | 設定與密鑰管理 | 所有 API key / 帳密走環境變數或 `.env` + `python-dotenv`;設定檔用 YAML/TOML 分層(dev/paper/prod) | 密鑰**絕不進 Git**;prod 設定與 paper 設定物理隔離,避免測試單打進實盤 |
| 0.4 | 日誌框架 | 統一用 `structlog` 或 `loguru`;所有模組輸出結構化 JSON log,含 timestamp、module、level | 交易系統的 log 是事後鑑識的唯一依據,格式從一開始就要統一 |
| 0.5 | 資料庫選型與部署 | 時序資料:TimescaleDB 或 DuckDB + Parquet;關聯資料(主檔、交易紀錄):PostgreSQL;完成本地部署 | 從零開始建議 DuckDB + Parquet 起步(零維運),量大後再遷 TimescaleDB;不要一開始就上分散式 |
| 0.6 | 任務排程器 | 部署 APScheduler(輕量)或 Airflow/Prefect(重量);能跑 cron 式日/週/月任務 | 從零開始用 APScheduler + systemd 即可;Airflow 維運成本高,等管線超過 20 條再考慮 |
| 0.7 | 統一時區與交易日曆基礎 | 全系統內部一律 UTC 存放,顯示層才轉台北/紐約時間;引入 `pandas_market_calendars` | 台美混合最常見的 bug 就是時區;美股夏令時間切換要寫測試驗證 |
| 0.8 | CI 與測試骨架 | pytest 跑通;GitHub Actions 每次 push 自動跑測試 | 交易邏輯與風控函式必須有單元測試;資料處理函式用固定樣本資料做回歸測試 |

## M1. 日常資料收集與處理(日/週/月)

### M1-A 資料源接入

| # | 任務 | 完成目標 | 注意事項 |
|---|------|---------|---------|
| 1.1 | 台股日線資料接入 | 從 TWSE/TPEx 官方 API 或 FinMind 抓上市+上櫃日 OHLCV,寫入資料庫 | TWSE 官網有請求頻率限制;上櫃(TPEx)是獨立來源,別漏;注意全額交割股與處置股標記 |
| 1.2 | 美股日線資料接入 | 接 Polygon / Alpaca / yfinance(擇一為主、一為備援),抓 OHLCV | yfinance 只能當原型驗證,不可當生產資料源(會被限流、資料會改);正式上線前換付費源 |
| 1.3 | 台指期/選擇權資料接入 | 從期交所(TAIFEX)每日下載期貨與選擇權日資料(含各履約價 OHLCV、未平倉) | 期交所提供 CSV 下載;選擇權資料量大,先定義只存哪些月份與 delta 範圍 |
| 1.4 | 美股期權資料接入 | 接入期權鏈快照(如 Polygon options 或券商 API) | 美股期權全鏈資料極大,先只存目標標的;歷史期權資料很貴,評估是否真的需要 |
| 1.5 | 除權息與公司行動資料 | 台股:抓除權息日程與配股配息;美股:抓 splits/dividends | 這是還原股價正確性的前提;台股「減資」是特殊事件,還原邏輯要另外處理 |
| 1.6 | 基本面資料接入 | 台股:月營收、季財報(公開資訊觀測站/FinMind);美股:財報數據(如 FMP、SEC EDGAR) | **point-in-time 是命門**:必須記錄「這筆財報是何時公佈的」,不能用財報期間當可用時間,否則回測全是前視偏差 |
| 1.7 | 三大法人與籌碼資料(台股) | 每日抓三大法人買賣超、融資融券、借券 | 這是台股特有 alpha 來源;注意法人資料公佈時間是盤後,回測時要對齊到 T+1 才可用 |
| 1.8 | 總經與利率資料 | 接 FRED API 抓利率、CPI、就業等;台灣抓央行/主計處數據 | 總經數據有「初值→修正值」問題,存 vintage(發佈版本)而非只存最新值 |

### M1-B 資料管線與品質

| # | 任務 | 完成目標 | 注意事項 |
|---|------|---------|---------|
| 1.9 | 統一資料 schema 設計 | 定義 OHLCV、基本面、籌碼等各類表的欄位規格文件;台美股共用同一 schema | 證券代碼要設計全域唯一 ID(如 `2330.TW`, `AAPL.US`),處理台股代碼重用問題 |
| 1.10 | 每日 ETL 管線 | 排程器每日收盤後自動:抓取→驗證→清洗→入庫→發完成通知 | 管線必須**冪等**(重跑不會重複寫入);台股 15:00 後、美股收盤後(台北清晨)各跑一次 |
| 1.11 | 資料品質驗證層 | 每批資料入庫前自動檢查:缺值、價格跳動超過閾值、成交量為零、高低價邏輯錯誤(high < low) | 驗證失敗要擋下並告警,而不是默默入庫;用 `pandera` 或 `great_expectations` |
| 1.12 | 還原股價計算模組 | 產出前復權/後復權價格序列,經除權息與減資調整 | 回測用後復權;訊號計算當下價格用原始價;兩者用途不同,欄位分開存 |
| 1.13 | 缺漏回補機制 | 提供 CLI 指令可指定日期區間回補任一資料源 | 資料源偶爾漏資料是常態;回補要走同一條驗證管線 |
| 1.14 | 存活者偏差處理 | 資料庫保留下市股票的歷史資料;維護「每日可交易股票清單」 | 只用現存股票回測會高估報酬 2~4%/年;FinMind 有下市股資料,從第一天就要收 |
| 1.15 | 週/月級彙總任務 | 週線/月線自動從日線彙總;月營收公佈後自動觸發相關更新 | 彙總以交易日曆為準,不是日曆週;台美假日不同,分開處理 |

## M2. 公司資料庫(對應「公司資料介紹」)

| # | 任務 | 完成目標 | 注意事項 |
|---|------|---------|---------|
| 2.1 | 公司主檔(master data) | 建立公司基本表:代碼、名稱、產業分類、上市日、市值分級、交易所 | 產業分類同時存官方分類(TSE 產業別/GICS)與自訂分類兩套 |
| 2.2 | 公司描述與業務摘要 | 每家公司一份結構化描述:主營業務、產品線、營收比重、主要客戶 | 台股從年報與公開資訊觀測站取得;可用 LLM 從年報自動抽取,但抽取結果要抽樣人工核對 |
| 2.3 | 財務指標計算層 | 從原始財報自動算出 ROE、毛利率、營益率、負債比等標準指標 | 指標公式寫成可測試的純函式;台美會計準則(IFRS vs US GAAP)差異要註記 |
| 2.4 | 公司資料查詢介面 | 一個函式/CLI:輸入代碼,輸出該公司完整快照(基本面+籌碼+近期事件) | 這是日常研究的高頻入口,回應速度要快(<1 秒) |
| 2.5 | 自動公司報告產生器 | 輸入代碼自動產出一頁式 Markdown/PDF 報告(業務+財務趨勢圖+估值) | 先做 Markdown 版,格式穩定後再加圖表;這是與 AI 協作的天然接口 |
| 2.6 | 同業比較模組 | 輸入代碼,自動列出同業並排比較估值與財務指標 | 「同業」定義用你的自訂分類,官方分類常常不準(例如台股「其他電子」) |

## M3. 供應鏈搜尋與整理

| # | 任務 | 完成目標 | 注意事項 |
|---|------|---------|---------|
| 3.1 | 供應鏈資料模型設計 | 定義 graph schema:節點=公司,邊=供應/客戶/競爭關係,邊帶屬性(營收占比、資料來源、時間戳) | 用 NetworkX + SQLite 起步即可,不用一開始上圖資料庫;每條關係必附來源與日期 |
| 3.2 | 公開來源供應鏈抽取 | 從年報「主要進銷貨對象」、新聞、產業報告抽取關係,半自動入庫 | 台股年報只揭露占比 >10% 的客戶且常匿名(「客戶A」);要建立推斷規則並標記信心等級 |
| 3.3 | LLM 輔助關係抽取管線 | 餵新聞/年報段落給 LLM,輸出結構化(公司A, 關係, 公司B, 證據句) | LLM 會幻覺出關係;**每筆必須帶原文證據句**,無證據不入庫 |
| 3.4 | 供應鏈查詢與視覺化 | 輸入公司,輸出上下游關係圖(2 層內)與清單 | 視覺化用 pyvis/graphviz 即可;重點是查詢正確,不是畫得漂亮 |
| 3.5 | 供應鏈事件傳導分析 | 輸入事件(如某公司砍單),列出受影響的上下游候選名單 | 這是供應鏈資料的變現點:接到 M9 事件行事曆與 M6 策略研究 |
| 3.6 | 關係時效性維護 | 每季 review 機制:超過 N 個月未確認的關係降級標記 | 供應鏈會變,舊關係不清理會產生錯誤訊號 |

## M4. 因子庫與研究框架

| # | 任務 | 完成目標 | 注意事項 |
|---|------|---------|---------|
| 4.1 | 因子計算框架 | 統一介面:輸入(universe, 日期區間)→輸出因子值矩陣;因子邏輯一檔一因子 | 所有因子計算只能用「當時已知」的資料(point-in-time);這是框架層要強制的,不能靠自覺 |
| 4.2 | 基礎因子實作(第一批) | 完成 10~15 個經典因子:動能、反轉、波動率、規模、價值(PB/PE)、月營收 YoY、法人買賣超 | 先求正確再求多;每個因子附單元測試與已知期間的預期值 |
| 4.3 | 因子檢定框架 | 自動產出:IC/RankIC 時序、分位數組合報酬、多空報酬、換手率 | IC 要看時序穩定性不是只看平均;分位數回測要含交易成本 |
| 4.4 | 因子衰退監控 | 每月自動重跑所有因子近 N 年 rolling IC,產出衰退報告 | 因子會死;衰退監控是決定策略下架的依據之一 |
| 4.5 | 實驗記錄系統 | 每次回測自動記錄:git commit hash、參數、資料版本、結果指標,存入實驗資料庫 | 用 MLflow 或自建 SQLite 皆可;**沒記錄的實驗等於沒做過**,這是防過擬合的制度防線 |
| 4.6 | 研究知識庫 | 建立 Markdown 知識庫:每個嘗試過的想法一頁(假設、做法、結果、失敗原因) | 失敗紀錄比成功紀錄有價值;這也是餵給 AI 的最佳 context |

## M5. 回測引擎

| # | 任務 | 完成目標 | 注意事項 |
|---|------|---------|---------|
| 5.1 | 回測引擎選型決策 | 評估 vectorbt(向量化、快)vs backtrader(事件驅動)vs 自建;寫決策文件 | 建議:因子研究用向量化,含期權/複雜下單邏輯用事件驅動;兩者並存是常態 |
| 5.2 | 交易成本模型 | 實作台股(手續費+證交稅 0.3%/期貨另計)與美股(佣金+SEC fee)成本函式 | 台股當沖稅率減半、ETF 稅率 0.1% 等特例要涵蓋;成本模型錯誤是回測失真主因之一 |
| 5.3 | 滑價模型 | 至少實作:固定 bps、按成交量比例(participation rate)兩種滑價模型 | 小型股滑價遠大於固定假設;先保守估計,等 paper trading 有實測數據再校準 |
| 5.4 | 部位與訂單撮合邏輯 | 事件驅動回測中正確處理:開收盤價成交假設、漲跌停無法成交、零股/整股 | 台股漲跌停 10% 且鎖死常見,**漲停買不到、跌停賣不掉**必須模擬,否則反轉策略回測全是假的 |
| 5.5 | Walk-forward 驗證框架 | 標準化流程:訓練窗/驗證窗滾動,自動產出 out-of-sample 績效 | 禁止用全樣本調參後報告全樣本績效;WF 是預設流程不是選配 |
| 5.6 | 績效指標模組 | 統一計算:年化報酬、Sharpe、Sortino、MDD、Calmar、勝率、盈虧比、月報酬分佈 | 指標算法寫成獨立套件,回測/paper/實盤三處共用同一份程式碼,避免三處算法不一致 |
| 5.7 | 回測報告產生器 | 一鍵產出 HTML/Markdown 報告:淨值曲線、回撤圖、月報酬熱圖、持股分析 | 用 quantstats 起步即可;報告自動存入 M4.5 實驗記錄 |
| 5.8 | 期權回測支援 | 期權部位的損益計算、到期結算、希臘值追蹤(至少 delta) | 期權回測資料品質是最大瓶頸;先從簡單策略(covered call、賣方價差)驗證引擎正確性 |

## M6. 策略開發與組合管理

| # | 任務 | 完成目標 | 注意事項 |
|---|------|---------|---------|
| 6.1 | 策略介面標準化 | 定義抽象基類:`generate_signals()` → `target_positions()`;所有策略實作同一介面 | 介面統一才能讓回測、paper、實盤共用同一份策略碼;這是「回測即實盤」的前提 |
| 6.2 | 第一支基準策略 | 完成一支簡單策略(如台股月營收動能或美股 12-1 動能)走完全流程:回測→WF→報告 | 目的不是賺錢,是打通全管線;選規則簡單、換手低的策略當管線試金石 |
| 6.3 | 策略參數管理 | 策略參數全部外部化(YAML),與程式碼分離;參數版本入實驗記錄 | 硬編碼參數是災難源頭;參數檔要能區分回測/paper/prod 三套 |
| 6.4 | 組合層資金配置模組 | 實作至少兩種配置法:等權重、波動率倒數加權;輸入各策略訊號,輸出整體目標部位 | 策略間相關性要監控;新策略上線先給小額度,依實盤表現漸進加碼 |
| 6.5 | 策略生命週期管理 | 定義並文件化:研究中→paper→小額實盤→正式→觀察→下架 的晉升與退場條件 | 退場條件要**事前**寫死(如 rolling Sharpe < X 或 MDD 超標),避免虧損時捨不得關 |
| 6.6 | 機會掃描器 | 每日收盤後自動掃描全市場,依各策略條件輸出候選清單與排名 | 這對應你原始需求的「機會尋找」;輸出接到 M10 通知系統 |

## M7. 風險管理模組(獨立於策略)

| # | 任務 | 完成目標 | 注意事項 |
|---|------|---------|---------|
| 7.1 | 風控規則引擎骨架 | 獨立服務/模組:輸入(目標下單、當前部位、帳戶狀態)→輸出(放行/縮量/拒絕) | **風控必須是下單的必經關卡**,不是策略自律;架構上策略碰不到下單 API,只有風控層碰得到 |
| 7.2 | 部位限制規則 | 實作:單一標的上限(%資產)、單一產業上限、總槓桿上限、單日新增部位上限 | 限制值寫在設定檔;台股要另設「處置股/全額交割股禁入」規則 |
| 7.3 | 停損與回撤熔斷 | 實作:單策略 MDD 熔斷、帳戶層日虧損熔斷(觸發即全面停新單+告警) | 熔斷後的恢復必須**人工確認**,不能自動恢復;熔斷邏輯要有單元測試 |
| 7.4 | 保證金與資金監控 | 期貨/選擇權保證金即時追蹤,維持率低於閾值提前告警 | 台指期保證金調整(期交所會臨時調)要能自動更新;預留緩衝別貼著維持率跑 |
| 7.5 | 曝險彙總儀表 | 即時彙總:各市場/產業/策略的多空曝險、delta 總和(含期權) | 期權曝險用 delta 等效換算;台美部位要換匯後用同一幣別彙總 |
| 7.6 | 風控事件日誌 | 所有風控攔截/縮量/熔斷事件獨立記錄,含觸發規則與當下快照 | 這份日誌是檢討風控參數鬆緊的唯一依據 |

| 8.3 | 訂單管理系統(OMS)核心 | 訂單狀態機:建立→送出→部分成交→完全成交/取消/拒絕;所有狀態變化入庫 | 狀態機要處理「送出後斷線不知道有沒有成交」的情境:重連後先查詢對帳再決定重送 |
| 8.4 | Paper trading 模式 | OMS 支援 paper 模式:用即時行情模擬撮合,不碰真錢 | Paper 與實盤走**同一份程式碼**,只切換執行後端;這是驗證訊號延遲與下單邏輯的關鍵階段 |
| 8.5 | 下單執行邏輯(第一版) | 實作市價單、限價單送單與追價邏輯(限價未成交 N 秒後調整) | 從最簡單的邏輯開始;台股集合競價時段(開盤前)與零股盤的規則不同要區分 |
| 8.6 | 每日自動對帳 | 每日收盤後:系統帳 vs 券商 API 回報的部位/現金逐筆核對,不符即告警 | 對帳不符是嚴重事件,預設行為是**停新單**直到人工排除;這是資金安全的底線 |
| 8.7 | 斷線重連與災難復原 | API 斷線自動重連;重啟後能從資料庫還原完整狀態(部位、掛單) | 系統狀態的唯一真相來源是資料庫+券商查詢,不是記憶體;重啟恢復流程要實際演練 |
| 8.8 | 交易成本分析(TCA) | 每筆成交記錄:訊號價 vs 實際成交價,自動統計滑價分佈 | TCA 數據回饋給 M5.3 滑價模型校準;這是回測與實盤對得起來的橋樑 |

## 全域注意事項(貫穿所有模組)

1. **Point-in-time 紀律**:任何資料入庫都要記「何時可知」,這是回測可信度的第一原則。
2. **回測/Paper/實盤同碼**:策略與風控邏輯三環境共用,只切換資料源與執行後端。
3. **失敗預設安全(fail-safe)**:任何不確定狀態(斷線、對帳不符、資料延遲)的預設行為都是「停止新動作+告警」,不是繼續跑。
4. **先窄後寬**:每個模組先支援台股日線一條路打通,再擴美股、再擴期權;水平擴張永遠比垂直打通容易。
5. **人工介入介面**:每個自動化環節都要留「一鍵暫停」與「手動覆寫」的口,並記錄(M11.4)。
